# Developer Guide

This document explains the architecture, development practices, and how to reproduce or extend the project.

## Project overview

- Purpose: provide sentiment analysis for news articles using a small ensemble (VADER, TextBlob, Transformer).
- Intended consumers: Marketing department (end users) and developers who will maintain and improve the code.

## Repository layout

- `app.py` — FastAPI web application and websocket flow used for interactive demos.
- `app.py` — FastAPI web application and packaging entrypoint for the Windows executable.
- `cli.py` — Command-line runner for offline testing and automation.
- `src/` — Core code modules:
  - `news_fetcher.py` — NewsAPI integration and article text extraction.
  - `sentiment.py` — Model wrappers for VADER, TextBlob, and Transformer.
  - `reporting.py` — Helpers to save results as CSV/JSON.
  - `main.py` — Small local runner for development/testing.
- `templates/` — Frontend bundles (HTML/JS/CSS) used by the web app.
- `data/reports/` — Stored output files generated by the pipeline.

## Coding conventions

- Use clear, descriptive function names and module-level docstrings.
- Prefer small functions with single responsibility.
- Add docstrings (Google-style or reST) to public functions — include Args/Returns/Exceptions where applicable.
- Use `logging` instead of `print()` for messages.

## Running locally for development

1. Create and activate virtual environment:

```bash
python -m venv .venv
.venv\Scripts\activate   # Windows
source .venv/bin/activate  # macOS / Linux
```

2. Install dependencies (use pinned file for reproducible results):

```bash
pip install -r requirements.pinned.txt
```

3. Run the CLI for a quick test:

```bash
python cli.py --keyword Tesla --num-articles 10
```

4. Run the web app (development):

```bash
python app.py
```

## Packaging into an executable (Windows)

We use `pyinstaller` to create a single-file executable from `app.py`.

1. Install PyInstaller in your environment:

```bash
pip install pyinstaller
```

2. Build the EXE from the project root:

```bash
pyinstaller --noconfirm --clean IA_Sentiment.spec
```

3. The built EXE will be in `dist/IA_Sentiment.exe`.

Notes:
- The `--add-data` argument ensures bundled templates and data end up inside the package. On non-Windows platforms the separator is `:` instead of `;`.
- Test the EXE on a clean machine to ensure required redistributables (e.g., MSVC runtimes) are present.

## Reproducibility and models

- Transformer model: `cardiffnlp/twitter-roberta-base-sentiment-latest` (logged on first load by the app). Record and pin model identifiers in releases.
- Use `requirements.pinned.txt` for deterministic package versions.

## Tests and CI

- Add unit tests in `tests/` and use `pytest` to run them.
- Add a GitHub Actions workflow to run `pytest` and code linters on PRs.

## Extending the project

- To add a new model, create a new function in `src/sentiment.py` that follows the same interface as the existing `analyze_sentiment_*` functions and update `analyze_sentiment_ensemble` to include it.
- To add additional metadata to reports, extend the dictionary appended to `results` and update `src/reporting.py` to persist the extra fields.

## Contact / Handover notes

- Keep `NEWS_API_KEY` out of committed files. Use `.env` or environment secrets in CI.
- Document any manual steps (model downloads, large-file caching) in this guide.

## Network / Port behavior and troubleshooting

- The application prefers port `8000` by default but will try the next free port automatically if the preferred port is already in use. You can request a specific port with the `PORT` or `APP_PORT` environment variable.
- When running the packaged EXE run it from a terminal to observe logs. If the EXE exits immediately, look for a bind error — another process may already be running. Use `tasklist`/`taskkill` to inspect and stop prior instances.
- If the transformer model fails on first run, pre-download it while developing to cache weights and reduce first-run failures during packaging.
